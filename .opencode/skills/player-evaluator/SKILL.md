# Player Evaluator Skill

多视角游戏概念评估，从玩家角度而非设计者假设出发。

---

## ⚠️ EXECUTION RULES

1. **PARALLEL EXECUTION** — 多个评估必须并发执行
2. **PLAYER VOICE** — 用第一人称"我"表达玩家视角
3. **ACTIONABLE FEEDBACK** — 每个问题必须附带建议
4. **CONSENSUS FOCUS** — 优先报告多视角共识问题

---

## Quick Actions

| 命令 | 行动 |
|------|------|
| `/player-evaluator` | 评估当前游戏概念 |
| `/player-evaluator profiles` | 查看可用玩家视角 |
| `/player-evaluator add [profile]` | 添加特定视角评估 |

## Phase 3: Launch Parallel Evaluations

### ⚡ 并行执行 (强制)

**FOR EACH selected profile**, 同时启动:

```typescript
task(
  category="deep",
  load_skills=["game-designer"],
  run_in_background=true,
  description="Evaluate as [Profile ID]",
  prompt="
你正在以特定玩家类型评估一个游戏概念。

## 你的玩家档案

**类型**: [Bartle 类型]
**经验水平**: [休闲/核心/硬核]
**主要动机**: [主要游戏动机]
**特征**: [关键心理特征]

## 游戏概念

[从 GDD 提取相关部分]

## 你的任务

从**你的玩家视角**评估这个概念。回答：
1. "我会玩这个游戏吗？为什么？"
2. "对我来说，什么会让它更好？"

## 评分维度 (1-5分)

1. **有趣性 (Fun Factor)**: 内在享受感
2. **参与度 (Engagement)**: 我会持续玩吗？
3. **学习曲线 (Learning Curve)**: 我能快速理解吗？
4. **动机循环 (Motivation Loop)**: 进度是否吸引我？
5. **目标玩家匹配 (Target Player Fit)**: 这适合像我这样的人吗？

## 输出格式

用第一人称"我"表达:

### 整体印象 (2-3句话)
### 维度评分
- 有趣性: X/5 — [理由]
- 参与度: X/5 — [理由]
- 学习曲线: X/5 — [理由]
- 动机循环: X/5 — [理由]
- 目标匹配: X/5 — [理由]

### 我喜欢的 (2-3点)
### 我担心的 (2-3点 + 具体建议)
"
)
```

**必须并行启动所有视角** — 不要顺序执行。

## Phase 4: Collect Results

**等待所有后台任务完成**。系统会发送 `<system-reminder>` 通知。

**For each completed task**:
- 调用 `background_output(task_id="...")`
- 解析维度评分
- 提取玩家声音反馈
- 识别共识问题

### 聚合算法

```
共识问题 = 被 2+ 视角提及的问题
严重问题 = 被 3+ 视角提及 或 2+ 视角标记为"严重"
独特洞察 = 仅被 1 个视角提及
```

## What This Skill Must NOT Do

- ❌ 做最终设计决策 (那是 creative-director 的职责)
- ❌ 自动实施更改
- ❌ 评估代码或技术实现 (那是 code-review 的职责)
- ❌ 不经询问就覆盖用户选择的视角

---

## Collaboration Protocol

**支持的 skills**:
- `creative-director` — 解决评估中发现的支柱冲突
- `game-designer` — 实施建议的改进
- `design-review` — 更改后的后续验证

**委托**:
- 玩家心理学框架 → 作为知识库加载
- 详细评估 → 并行 deep agents (后台)
- 报告综合 → 由本 skill 完成 (编排)
