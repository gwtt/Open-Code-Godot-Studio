# 美术圣经编写技能

分阶段引导创建完整的视觉规范文档。

---

## ⚠️ 执行规则

1. **每阶段一问** — 每个阶段只问一个问题，然后等待
2. **先读后问** — 在提问前先读取所有相关文档
3. **立即写入** — 每个部分批准后立即写入文件
4. **不跳阶段** — 按顺序完成，不跳过

---

## 前置条件

读取 `design/gdd/game-concept.md`。如果不存在：
> "未找到游戏概念文档。请先运行 `/brainstorm` — 美术圣经在游戏概念批准后编写。"

从 game-concept.md 提取：
- 游戏标题（工作标题）
- 核心幻想和电梯演讲
- 游戏支柱（全部）
- **视觉身份锚点**（如果存在于 brainstorm Phase 4 的输出中）
- 目标平台（如果有）

## Phase 1: 确定范围

使用 `question` 工具询问：

**问题 1 — 范围**：
```
今天需要编写哪些部分？

A) 完整圣经 — 全部 9 个部分
B) 视觉核心（第 1-4 部分）
C) 仅资产标准（第 8 部分）
D) 继续 — 填充缺失部分
```

**问题 2 — 参考**：
```
是否有定义视觉方向的参考游戏、电影或艺术作品？
[自由文本输入]
```

如果 game-concept.md 有视觉身份锚点：
> "发现 brainstorm 中的视觉身份锚点：'[锚点名称] — [规则]'。将以此作为美术圣经的基础。"

---

## Phase 2: 视觉基础（第 1-4 部分）

这四个部分定义核心视觉语言。**所有其他部分都源自它们。**

### 第 1 部分：视觉身份声明

**目标**: 一行视觉规则 + 2-3 条支持原则，解决视觉歧义。

如果存在视觉锚点，询问：
- "直接基于此锚点构建？"
- "扩展前先修订？"
- "从头开始新选项？"

**关键问题**：
```
为这个游戏起草视觉身份声明：
1. 一行视觉规则，能解决任何视觉决策歧义
2. 2-3 条支持原则，每条有设计测试（"当 X 模糊时，此原则说选择 Y"）
3. 每条原则必须服务于特定的游戏支柱
```

批准后立即写入文件。

---

### 第 2 部分：情绪与氛围

**目标**: 按游戏状态定义情感目标。

为每个主要游戏状态定义：
- 主要情绪/氛围目标
- 光线特征（时间、色温、对比度）
- 氛围描述词（3-5 个形容词）
- 能量级别（激烈/平稳/沉思等）

**关键问题**：
```
为每个游戏状态定义情绪和氛围目标：
- 具体的情感目标
- 光线特征（暖/冷、高/低对比、时间方向）
- 至少一个承载情绪的视觉元素
- 每个游戏状态必须视觉上区分明显
```

---

### 第 3 部分：形状语言

**目标**: 让游戏世界视觉连贯的几何词汇。

覆盖：
- 角色剪影哲学（缩略图大小可读性？每个原型的区分特征？）
- 环境几何（角度/曲线/有机/几何 — 哪种主导？为什么？）
- UI 形状语法（UI 回应世界美学，还是独立的 HUD 语言？）
- 主角形状 vs 辅助形状（什么吸引眼球，什么退后？）

---

### 第 4 部分：色彩系统

**目标**: 完整、可生产的配色系统。

覆盖：
- 主色板（5-7 色，含角色 — 不仅是 hex 代码，而是每种颜色在这个世界的意义）
- 语义色使用（红色传达什么？金色？蓝色？白色？建立色彩词汇）
- 每区域色温规则
- UI 色板（可能与世界色板不同）
- 色盲安全：哪些语义色需要形状/图标/声音备份

---

## Phase 3: 制作指南（第 5-8 部分）

这些部分将视觉身份转化为具体的制作规则。

### 第 5 部分：角色设计方向

- 玩家角色的视觉原型
- 每种角色类型的区分特征规则
- 表情/姿态风格目标
- LOD 哲学

### 第 6 部分：环境设计语言

- 建筑风格
- 纹理哲学（手绘 vs PBR vs 风格化）
- 道具密度规则
- 环境叙事指南

### 第 7 部分：UI/HUD 视觉方向

- UI 视觉风格 — 沉浸式 vs 屏幕空间 HUD
- 字体方向
- 图标风格
- UI 动画感觉

**注意**: 如果视觉方向与可读性/无障碍需求冲突，明确指出并让用户决定。

### 第 8 部分：资产标准

- 文件格式偏好
- 命名规范方向
- 纹理分辨率层级
- LOD 级别期望
- 导出设置哲学
- 多边形预算（每资产类别）
- 纹理内存限制

---

## Phase 4: 参考方向（第 9 部分）

**目标**: 策划的参考集，明确借鉴什么和避免什么。

提供 3-5 个参考来源：
- 名称
- 明确要借鉴的视觉元素
- 明确要避免或偏离的

参考应该是增量的 — 没有两个参考应该指向完全相同的方向。

---

写入文件：`design/art/art-bible.md`

```markdown
# 美术圣经 — [游戏名称]

> **创建日期**: [日期]
> **基于**: design/gdd/game-concept.md
> **状态**: 初稿完成 / 已批准

## 目录

1. 视觉身份声明
2. 情绪与氛围
3. 形状语言
4. 色彩系统
5. 角色设计方向
6. 环境设计语言
7. UI/HUD 视觉方向
8. 资产标准
9. 参考方向

---

[各部分内容...]
```

## 协作协议

每个部分遵循：**问题 → 选项 → 决定 → 草案 → 批准 → 写入文件**

- 每部分写入后立即批准
- 不要批量写入
- 所有冲突明确指出
- 美术圣经是约束文档：每部分应该感觉上有意义地缩小解决方案空间
