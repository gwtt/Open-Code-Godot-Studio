## 团队概况

### 典型配置

```
┌─────────────────────────────────────────────────────────┐
│                    2人小队                               │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   👨‍💻 程序 (1人)              🎨 美术 (1人)            │
│   ├── 技术决策                ├── 视觉风格              │
│   ├── 代码实现                ├── 资产制作              │
│   ├── 系统架构                ├── UI设计                │
│   └── Bug修复                 └── 动画/特效             │
│                                                         │
│              共同负责：游戏设计、进度规划                 │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 核心优势

- **沟通成本低**: 直接交流，无需文档传递
- **决策快**: 两人共识即可推进
- **灵活调整**: 随时改变方向

### 主要挑战

- **技能互补**: 程序不懂美术，美术不懂代码
- **时间管理**: 需要协调制作进度
- **规范一致**: 命名、结构需要统一

> ⚠️ **关键决策**: 不同类型的游戏需要不同的资源分配策略

### 三种游戏类型

```

├─────────────────────────────────────────────────────────────────────────┤

│  🎨 美术驱动型                    ⚖️ 平衡型                    👨‍💻 程序驱动型  │
│  ┌─────────────┐               ┌─────────────┐               ┌─────────────┐ │
│  │ ████████████│               │ ████████████│               │ ████████████│ │
│  │ ████████████│               │ ████████████│               │ ████████████│ │
│  │ ████████████│               │ ████████████│               │ ░░░░░░░░░░░░│ │
│  │ ████████████│               │ ░░░░░░░░░░░░│               │ ░░░░░░░░░░░░│ │
│  │ ░░░░░░░░░░░░│               │ ░░░░░░░░░░░░│               │ ░░░░░░░░░░░░│ │
│  └─────────────┘               └─────────────┘               └─────────────┘ │
│   美术 70% 程序 30%              美术 50% 程序 50%             美术 30% 程序 70% │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 类型判定指南

| 问题 | 美术驱动型 | 平衡型 | 程序驱动型 |
|------|-----------|--------|-----------|
| 核心乐趣来源 | 视觉体验、氛围 | 玩法+视觉平衡 | 机制、系统深度 |
| 玩家留存原因 | "画面太美了" | "好玩又好看" | "策略性很强" |
| 美术资产量 | 大量独特资产 | 中等，复用高 | 少量，功能性 |
| 代码复杂度 | 相对简单 | 中等 | 高度复杂 |
| 典型游戏 | 视觉小说、解谜、银河城 | 平台跳跃、ARPG、塔防 | Roguelike、策略、模拟 |

### 美术驱动型游戏

**特点**: 美术是核心竞争力，程序服务于美术

| 游戏类型 | 美术占比 | 美术重点 | 程序重点 |
|----------|----------|----------|----------|
| **视觉小说** | 80% | 立绘、背景、CG | 对话系统、存档 |
| **解谜游戏** | 70% | 场景氛围、动画 | 交互逻辑、关卡 |
| **银河城** | 65% | 地图、角色、特效 | 移动、战斗、地图 |
| **休闲手游** | 70% | UI、角色、动效 | 社交、支付、广告 |

**资源分配建议**:
```
时间分配:
├── 美术: 60-70%
│   ├── 角色设计: 30%
│   ├── 场景/背景: 25%
│   ├── UI/特效: 25%
│   └── 动画: 20%
│
└── 程序: 30-40%
    ├── 核心系统: 50%
    ├── UI实现: 25%
    └── 优化/测试: 25%
```

**技能使用侧重**:
```bash
# 美术主导
/art-bible → 详细视觉规范
/asset-spec → 大量资产规格
/art-coordinator → 频繁协调

# 程序辅助
/godot-gdscript → 快速原型（可选GDScript）
/prototype-mode → 快速验证美术需求
```

### 平衡型游戏

**特点**: 美术和程序同等重要，紧密配合

| 游戏类型 | 美术占比 | 美术重点 | 程序重点 |
|----------|----------|----------|----------|
| **平台跳跃** | 50% | 角色、关卡、UI | 物理、关卡、存档 |
| **ARPG** | 50% | 角色、特效、场景 | 战斗、技能、AI |
| **塔防** | 50% | 塔、敌人、特效 | 波次、AI、平衡 |
| **横版射击** | 50% | 飞船、敌人、背景 | 子弹、碰撞、关卡 |

**资源分配建议**:
```
时间分配:
├── 美术: 50%
│   ├── 角色/单位: 35%
│   ├── UI/HUD: 25%
│   ├── 场景/背景: 25%
│   └── 特效: 15%
│
└── 程序: 50%
    ├── 核心玩法: 40%
    ├── 系统: 30%
    ├── UI: 20%
    └── 优化: 10%
```

**技能使用侧重**:
```bash
# 双方紧密配合
/brainstorm → 共同参与
/design-system → 美术参与视觉需求
/asset-spec → 程序识别需求，美术规格化

# 程序
/godot-csharp → 主力开发
/code-review → 质量保证

# 美术
/art-bible → 风格定义
/art-coordinator → 持续协调
```

### 程序驱动型游戏

**特点**: 玩法机制是核心竞争力，美术服务于功能

| 游戏类型 | 美术占比 | 美术重点 | 程序重点 |
|----------|----------|----------|----------|
| **Roguelike** | 30% | 角色、UI、特效 | 生成、平衡、进度 |
| **策略游戏** | 30% | 地图、单位、UI | AI、系统、网络 |
| **模拟经营** | 35% | 建筑、UI、角色 | 系统、AI、数据 |
| **多人对战** | 30% | 角色、特效、UI | 网络、同步、反作弊 |

**资源分配建议**:
```
时间分配:
├── 美术: 30-35%
│   ├── UI/HUD: 40%  ← 功能性优先
│   ├── 角色/单位: 30%
│   ├── 特效: 20%
│   └── 场景: 10%
│
└── 程序: 65-70%
    ├── 核心系统: 45%
    ├── AI/算法: 25%
    ├── 数据/存档: 15%
    └── UI/网络: 15%
```

**技能使用侧重**:
```bash
# 程序主导
/technical-director → 架构决策
/godot-csharp → 复杂系统
/godot-gdextension → 性能关键模块

# 美术辅助
/art-bible → 简化规范
/asset-spec → 功能性资产
/art-coordinator → 批量协调
```

### 资源布局建议

#### 目录结构（按游戏类型）

```
美术驱动型:
assets/
├── characters/       # 角色（最多资源）
│   ├── player/
│   ├── npcs/
│   └── enemies/
├── backgrounds/      # 场景背景（大量）
├── effects/          # 特效
└── ui/               # UI

平衡型:
assets/
├── characters/       # 角色
├── environments/     # 环境
├── effects/          # 特效
├── ui/               # UI
└── audio/            # 音效

程序驱动型:
assets/
├── ui/               # UI（最重要）
├── effects/          # 特效
├── characters/       # 角色（简化）
└── tiles/            # 功能性瓦片
```

## 开发者控制台

> 💡 **核心理念**: 让程序和美术都能通过 OpenCode 控制开发流程

### 什么是开发者控制台？

开发者控制台是 **OpenCode 会话本身**，通过技能命令控制整个开发流程。

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        OpenCode 开发者控制台                              │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   输入命令                        执行结果                               │
│   ┌──────────────────────┐      ┌──────────────────────┐               │
│   │ /start               │  →   │ 团队配置、项目初始化   │               │
│   │ /brainstorm          │  →   │ 游戏概念文档          │               │
│   │ /art-bible           │  →   │ 美术圣经              │               │
│   │ /asset-spec          │  →   │ 资产规格文档          │               │
│   │ /godot-csharp        │  →   │ 代码指导              │               │
│   │ /sprint-plan         │  →   │ Sprint 计划           │               │
│   │ /producer            │  →   │ 进度报告              │               │
│   └──────────────────────┘      └──────────────────────┘               │
│                                                                         │
│   ═══════════════════════════════════════════════════════════════════   │
│                                                                         │
│   程序员控制台                    美术师控制台                           │
│   ├── /setup-engine              ├── /art-bible                         │
│   ├── /technical-director        ├── /asset-spec                        │
│   ├── /godot-specialist          ├── /art-coordinator                   │
│   ├── /code-review               └── 设计参与：/brainstorm              │
│   └── /dev-story                                                        │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 控制台权限

| 角色 | 可执行命令 | 说明 |
|------|-----------|------|
| **程序** | 全部命令 | 技术决策权 |
| **美术** | 美术相关 + 设计参与 | 视觉决策权 |

### 程序员控制台

**日常开发命令**:

```bash
# 开始工作
/start                          # 初始化会话
/sprint-status                  # 查看当前进度

# 开发
/dev-story story-001            # 实现 Story
/godot-csharp                   # C# 指导
/code-review                    # 代码审查

# 协调
/art-coordinator check          # 检查资产导入
/producer                       # 进度报告
```

```

│                                                             │

│ 显示: [Sprint 3] 进度 60%                                    │
```

### 控制台最佳实践

#### 1. 命名规范

```
资产命名 (程序 + 美术共同遵守):
├── 角色: char_[name]_[action]_[frame].png
│   例: char_knight_idle_01.png
│
├── UI: ui_[element]_[state].png
│   例: ui_button_hover.png
│
├── 特效: vfx_[name]_[variant].png
│   例: vfx_explosion_01.png
│
└── 音效: sfx_[action]_[variant].wav
    例: sfx_jump_01.wav
```

#### 2. 文件组织

```
design/
├── gdd/                    # GDD 文档
│   ├── game-concept.md     # 游戏概念
│   └── [system].md         # 系统设计
│
├── art/                    # 美术文档
│   └── art-bible.md        # 美术圣经
│
├── assets/                 # 资产规格
│   ├── asset-manifest.md   # 资产清单
│   └── specs/              # 规格文档
│
└── production/             # 生产文档
    └── sprints/            # Sprint 计划
```

#### 3. 版本控制

```bash
# 每日提交
git add .
git commit -m "feat: 完成敌人AI基础行为"

# 功能分支
git checkout -b feature/enemy-ai
# ... 开发 ...
git merge feature/enemy-ai
```

## 开发流程

### 完整开发周期

```
Week 1: 概念阶段
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

周一: /start → 团队检测 → 选择 "2人小队"
周二: /brainstorm → 探索游戏概念 → game-concept.md
周三: 审阅概念 → 双方确认方向
周四: /art-bible → 美术定义视觉风格 → art-bible.md
周五: /setup-engine → 程序配置引擎 → 技术偏好

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Week 2-3: 设计阶段
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Week 2:
├── /map-systems → 分解系统
├── /design-system → 编写 GDD (每个系统)
└── /review-all-gdds → 跨系统一致性检查

Week 3:
├── /asset-spec → 生成资产规格
```

### Q2: 美术不懂代码，如何理解技术约束？

**建议**: 程序用 `/asset-spec` 生成详细规格，美术按规格制作。
